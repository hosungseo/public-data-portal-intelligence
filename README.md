# Public Data Portal Intelligence

**Live page:** <https://hosungseo.github.io/public-data-portal-intelligence/>

![File-to-API reader preview](assets/file-to-api-reader.png)

공공데이터포털의 **파일데이터 중 API 전환 검토 우선순위가 높은 후보군**을 읽기 쉽게 보여주는 퍼블릭 리더입니다.

이 저장소가 답하려는 질문은 단순합니다.

> 공공데이터포털 파일데이터 가운데, 무엇을 먼저 API 전환 검토 대상으로 볼 것인가?

## 한 줄 정의

`public-data-portal-intelligence`는 파일데이터를 곧바로 공식 API로 바꾸는 엔진이 아니라, **에이전트와 사람이 공공데이터포털의 파일데이터/API 지형을 인식하고, API 전환 후보를 검토·설명·제안할 수 있게 만드는 transition-request / transition-intelligence layer**입니다.

## Agent behavior flow

에이전트 관점에서 이 저장소의 기본 행동은 3단계입니다.

### 1. 찾는다
- 이 주제에 이미 API가 있는지
- 파일데이터만 있는지
- API 전환 후보로 볼 가치가 있는지

즉, 첫 번째 의의는 "공공데이터포털에 이런 파일데이터와 API가 존재한다"는 사실을 에이전트가 인식할 수 있게 하는 것입니다.

### 2. 설명한다
에이전트는 주인에게 다음처럼 돌아갈 수 있습니다.
- 현재 이 데이터는 파일데이터로만 제공됩니다
- 이미 API가 존재합니다
- 우선 파일로 분석할 수 있습니다
- 이 데이터는 반복 수요가 높아 API 전환 검토 가치가 있습니다

### 3. 제안한다
필요하면 에이전트는 API 전환 요청까지 도울 수 있습니다.
- 전환 필요성 요약
- 왜 우선 검토 대상인지 설명
- 요청 사유 초안 제시
- 후속 검토 패킷 생성

즉, 이 저장소는 전환 실행기보다 **발견자 + 설명자 + 제안자** 역할에 더 가깝습니다.

## Transition request packet schema

이 저장소가 장기적으로 만들어야 할 핵심 산출물은 "전환 요청 패킷"입니다.

- 스키마 문서: [`docs/TRANSITION-REQUEST-PACKET.md`](docs/TRANSITION-REQUEST-PACKET.md)
- 목적: 후보를 설명 가능한 형태로 구조화하고, 사람이 후속 판단을 내릴 수 있게 만드는 것
- 성격: 자동 전환 엔진이 아니라 **검토/제안 패킷**

## Why this queue exists

파일데이터는 한 번 여는 순간마다 구조를 다시 파악해야 쓸 수 있습니다. 사람도 그렇고, 그 뒤에 오는 자동화 파이프라인도 그렇습니다. API는 그 단계를 한 번만 치르는 형태로 줄인 제공방식입니다. 방향은 분명합니다 — 궁극적으로 모든 파일데이터는 API로 제공되어야 합니다. 다만 한 번에 모두 전환하는 것은 불가능하므로, 수요가 반복되고 구조가 단순한 후보일수록 먼저 시작할 가치가 커집니다. 이 페이지는 어느 데이터에서 그 가치가 가장 크게 드러나는지를 현업이 스스로 훑어볼 수 있도록 돕습니다.

## Reviewed scope

공공데이터포털의 **파일데이터 88,166건 전체**가 검토 후보입니다. 사상적으로 궁극에는 모두 API여야 하므로 후보 자체에 demand 게이트는 두지 않았습니다. 그중 **다운로드 1,000건 이상 또는 국가중점데이터인 11,238건(전체의 12.8%)을 우선 후보**로 분류해 큐 상단에 정렬합니다. 나머지 76,928건도 동일 큐 안에 있고, 점수에 따라 자연스럽게 뒤로 배치됩니다.

## What this repo is for

이 프로젝트는 공공데이터포털의 전체 파일데이터를 공개하는 저장소가 아닙니다.
대신, 다음과 같은 **검토 큐**를 빠르게 읽기 위한 표면을 제공합니다.

- 수요가 이미 높은 파일데이터는 무엇인가
- 메타데이터가 충분해서 API 설계를 덜 추측적으로 시작할 수 있는 것은 무엇인가
- 어떤 제공기관이 API 전환 검토 후보를 많이 보유하고 있는가
- 파일 수요와 API형 수요가 동시에 관찰되는 후보는 어디인가

이 페이지는 자동 판정기가 아니라 **사람이 먼저 볼 검토 큐**입니다.

## What the page does

공개 페이지는 다섯 개 섹션으로 구성됩니다.

1. **히어로** — 선언문과 4개 KPI(검토 후보·우선 검토·즉시 검토 가능·교차수요 관찰). 첫 화면에서 규모와 질문을 같이 보여줍니다.
2. **우선 검토 대상 12건** — shortlist 카드. 각 카드를 펼치면 해당 행이 왜 우선순위에 올랐는지 근거 4가지와 "왜 먼저 검토?" 문구가 수치와 함께 나옵니다.
3. **전체 검토 큐 (11,238건)** — 검색·제공기관·검토 레인·포맷·정렬·페이지네이션. 아무 행이나 클릭해 펼치면 근거 요약·이유 4개·레인별 검토 사유를 확인할 수 있습니다.
4. **큐의 모습** — 결합 상태 분포 바, 주요 포맷·갱신 주기 칩, 기관 상위 5개.
5. **방법론과 공개 범위** — 접힘 패널. 사용된 자산, 이 페이지가 하지 않는 일, 공개 범위.

## Selection criteria

세 단계 기준이 후보 11,238건과 그 안의 우선순위를 결정합니다. (페이지 상단 "어떻게 골랐나" 섹션과 동일.)

**01 · 검토 후보** — 어떤 행이 후보가 되는가
- **공공데이터포털의 모든 파일형 데이터** (88,166건)
- 단, 목록·서비스 유형·데이터 형식에서 API형 패턴이 감지된 행(이미 API)은 제외

**02 · 우선 후보** — 어떤 후보가 우선으로 분류되는가
- 다운로드 수요 **1,000건 이상** **OR** **국가중점데이터(`국가중점여부 == "Y"`)**
- 전체 후보의 12.8% (11,238건)가 우선 후보. 큐 상단에 정렬

**03 · 검토 레인** — 어느 레인으로 분류되는가
- **즉시 검토 가능** — 응답 필드가 드러나고 메타정보 ≥ 4/5
- **교차수요 확인** — API형 신청이 이미 관찰됨
- **이력 재검증 필요** — 이용 이력이 결합되지 않음 (지원센터목록·메타는 있으나 이용이력 없음)
- **수요 우선** — 그 외, 다운로드 수요가 먼저 근거가 되는 경우

**04 · 우선 검토 점수** — 상위 12건 shortlist는 어떻게 정해지는가
- 종합 이용 신호 (다운로드 + API 신청 합계)
- 응답 필드 노출 여부
- 메타정보 풍부도 1–5
- 지원센터목록·메타·이용이력 3종 결합
- **국가중점데이터** 가산 (압도적 정책 가중)
- **포털 호스팅** 가산 (기관 호스팅보다 전환 마찰 ↓)
- **라이선스 제약** (변경금지·상업금지·비영리) 감산
- 같은 기관이 두 자리 이상 차지하지 않도록 배분

## Source joins

페이지는 공공데이터포털 위에 올라간 세 가지 카탈로그의 결합 상태를 `UMY` 형식으로 표시합니다.

| 축 | 라벨 | list_key | 제공기관 | 현재 적용 스냅샷 |
|---|---|---|---|---|
| **U** | 지원센터목록 | `15062804` | 공공데이터활용지원센터 | 2026-02-28 (최신본 2026-03-31 공개됨, 다음 갱신 시 적용 예정) |
| **M** | 메타 | `15121937` | 행정안전부 | 2025-05-31 |
| **Y** | 이용이력 | `15076332` | 행정안전부 | 2024-12-31 |

`UMY`는 세 소스 모두에 행이 있다는 뜻이고, `UM-`는 이용이력이 없고, `-MY`는 지원센터목록 쪽에 행이 없다는 뜻입니다. 모든 호출은 `api.odcloud.kr/api`를 거칩니다.

## Design

페이지는 다크 프리미엄 톤으로 설계되었습니다.

- **색** — 순검정 배경(`#000`), 네온 녹색 포인트(`#0af97a`)를 한 화면에 한두 곳만
- **타이포** — 한글 Pretendard Variable · 라틴 Inter · 숫자 SF Mono / JetBrains Mono
- **상호작용** — native `<details>`로 행 펼침, `prefers-reduced-motion` 지원, `:focus-visible` 녹색 아웃라인
- **반응형** — 1080px 기준에서 2열 레이아웃이 1열로, 640px 기준에서 KPI가 4→2열로 축소

## Data policy

이 repo는 의도적으로 **경량 reader asset만** 포함합니다.

Included:
- `index.html`
- `file-to-api.html`
- `file-to-api.js`
- `output/file_to_api_summary.{json,js}` — 집계·shortlist(상위 12건)·strongest(48건)
- `output/file_to_api_index.{json,js}` — 전체 10,045건 경량 인덱스(컬럼형, 한 행당 약 96바이트, gzip 후 약 246KB)

Not included:
- full merged master JSON
- raw source datasets
- internal scratch files
- intermediate analysis files
- 빌드 스크립트(`scripts/`)

공개 페이지는 이 lightweight summary/index asset을 브라우저에서 직접 읽습니다. `file_to_api_index`는 각 행의 title·provider·포맷·combo·플래그·수치만 담고, 근거 문장은 빌드 시 저장하지 않고 클라이언트에서 규칙으로 유도합니다. 빌드 스크립트와 완전히 동일한 규칙을 사용해 일관된 근거 문장을 생성합니다.

## Notes

- 방향은 모든 파일데이터의 API화로 분명하지만, 이 페이지는 **우선순위만** 자동 산출합니다. 각 데이터의 전환 시점·방식은 사람의 판단을 요구합니다.
- 실제 API 전환 여부는 스키마, 갱신 주기, 법적 제약, 운영 가능성 검토가 추가로 필요합니다.
- 공개 페이지는 전체 원천 공개보다 **검토 우선순위의 가독성**에 최적화되어 있습니다.
